# 标题

**ComfyUI Advanced Workflow with Custom LoRA Training (FLUX.1-dev)**

---

## 摘要

本报告呈现本项目已完成的技术实践：使用 `ComfyUI + Ostris AI Toolkit` 完成自训练 LoRA 风格迁移，并在统一条件下完成对比分析。实验全程固定单一底模 `FLUX.1-dev`，训练与推理保持同一语义族，不混用 Schnell。训练数据由 28 张日系魔幻主题图像及对应 caption 构成，触发词为 `jpfantasy_style`。`jpfantasy_style_stageA.safetensors` 作为核心训练产物用于方法说明与结果分析，流程文件边界见文末提交说明。
围绕作业三项要求，我完成了 A/B/C 三组实验：A 组用于比较 LoRA 开关与强度变化下的视觉差异，B 组用于比较 prompt 从短到长时的生成行为，C 组用于分析失败样例与当前局限（多人动态、文字可读性）。结果表明，在本次固定 seed、单轮出图条件下，LoRA 强度会显著影响风格与画面氛围，长 prompt 在同子组内通常会提高结果一致性；同时，手部结构、多体交互与复杂文字仍是主要短板。基于设备与时间约束，本项目在 Stage A 达到可用结果后终止 Stage B，以保证在课程周期内完成可复现、可解释的完整提交。

## 1. 引言（Introduction）

本项目采用“ComfyUI 高级工作流 + 自训练 LoRA”的完整闭环，并以可复现结果支撑技术报告，而非停留在在线工具体验层面。选择该路线的原因有三点：
第一，开源工作流具备较高可控性，能够显式管理模型、节点和参数，便于复现实验；
第二，LoRA 训练相比全量微调资源开销更低，更适合个人作业周期内完成；
第三，高级工作流实践要求强调完整技术闭环，本项目以“日系魔幻风格 LoRA”作为明确主题，能够形成可展示的训练-推理-对比链条。

本报告采用单底模路线：训练与推理统一使用 `FLUX.1-dev`，不引入第二 checkpoint 做横向模型对比。该决策既符合本项目约定，也使变量聚焦于“LoRA 强度”和“prompt 复杂度”。全文结构如下：第 2 节介绍环境与实验设置；第 3 节说明方法与对比协议；第 4–5 节对应作业前两条硬性要求；第 6 节讨论不足与展望；第 7 节给出结论。
从个人实践角度看，本项目最关键的工程取舍是“先完成可复现闭环，再考虑扩展规模”。在设备资源受限条件下，我先进行 5 步小规模测试确认训练链路可执行，再进入 Stage A 正式训练；在 Stage A 已满足实验需求后，集中完成 A/B/C 对比与结果分析，不再扩张训练规模。

## 2. 背景与实验环境（Background and Setup）

### 2.1 工具链与运行环境

本项目采用两套本地工具协同：
- 推理工具：`ComfyUI`（本机路径 `F:\AI_tool\ComfyUI`）
- 训练工具：`Ostris AI Toolkit`（路径 `F:\AI_tool\ai-toolkit`）

两者使用独立虚拟环境，避免依赖冲突。ComfyUI 负责所有 A/B/C 对比出图；Ostris 负责 LoRA 训练。Hugging Face 侧已完成账号认证并具备 `FLUX.1-dev` 访问权限。整体流程为：数据准备 -> caption 整理 -> Stage A 训练 -> ComfyUI 加载 LoRA 进行对比出图。

受设备性能与时间预算约束，本项目采用“分阶段降风险”策略：先进行 5 步小规模训练测试，确认配置、数据路径与训练链路可用后，再启动 Stage A 正式训练。具体保存策略与范围控制决策见 2.4 节。

### 2.2 模型路线与一致性约束

本项目唯一底模为 `black-forest-labs/FLUX.1-dev`。训练和推理均保持同一语义族，避免 dev/schnell 混用造成风格和行为不可控漂移。该一致性原则是本实验可信性的前提：若底模不一致，则 LoRA 的作用会与底模差异耦合，难以解释强度变化对应的视觉结果。

### 2.3 数据集与标签策略

训练集包含 28 张图像及对应 caption，目录为：
- 图像：`lora_model\data\train\images`
- 文本：`lora_model\data\train\captions`

命名采用 `jpfantasy_style_0001` 至 `jpfantasy_style_0028`。caption 采用“触发词 + 主体/场景要点 + 少量画风词”的压缩策略，触发词固定为 `jpfantasy_style`。与冗长 prompt 相比，压缩 caption 更利于保持训练目标聚焦，减少无关描述对风格学习的干扰。
训练图构建流程为：每个场景先在既定工作流下生成 20 张以上候选图，再人工筛选 1 张进入训练集，最终汇总为 28 张；场景目录中的参考图仅用于人工风格对照，不并入训练数据。

### 2.4 训练配置与产出

本次流程使用的核心权重来自 Stage A，文件为：
`lora_model\outputs\jpfantasy_style_stageA\jpfantasy_style_stageA.safetensors`。

Stage A 已完整跑通并保留训练日志，本报告全部结果基于 Stage A 产出完成。原计划的 Stage B 因设备性能与时间成本限制已取消且无后续安排：在低显存与降配设置下，Stage A 单次训练仍约需 2 小时，因此本次作业不再继续 Stage B。上述权重、训练配置与日志均作为流程说明材料，仅用于报告中的方法与边界说明，不作为附件提交。该取舍的核心原则是：在课程周期内优先保证结果可复现、结论可解释。
具体执行中，我先用 5 步测试确认链路正常，再进入 Stage A；为提高稳健性，Stage A 采用每 20 步保存一次的策略，最终获得可用于推理对比的 `jpfantasy_style_stageA.safetensors`。这一流程直接支撑了本报告的实验闭环与提交范围决策。


## 3. 方法与对比协议（Method）

### 3.1 统一基线

本次实验使用统一 ComfyUI 基线工作流（参数记录见 baseline 文档），并固定核心采样设置：`1024x1024`、`steps=20`、`sampler=euler`、`scheduler=simple`、`cfg=1.0`。
LoRA 侧固定 clip strength 为 `1.0`；A/B 组模型强度扫描为 `0.0 / 0.6 / 0.8 / 1.0`；C 组为 `0.0 / 1.0`。

### 3.2 三组实验设计

- **A 组（A_lora_on_off）**：多题材场景下测试 LoRA 开关与强度变化，主要回答“风格与观感差异如何体现”。

- **C 组（C_prompt_complexity）**：设置困难任务（双人打斗、夜市多招牌）用于局限分析，而非追求“最美图”。

## 4. 结果一：生成图视觉对比（对应要求 1）

### 4.1 A1/A2 的风格与氛围变化

在 A1（`A1_prompt_hero`，seed `781069391428044`）中，`0.0` 档更接近底模原生观感；当强度升至 `0.6` 后，目标风格特征开始明显增强；`0.8` 与 `1.0` 已较接近，但 `1.0` 常呈现更高亮度与更强体积感。
在 A2（`A2_prompt_city`，seed `558036329660600`）中，差异更具可描述性：本次实验下 `0.0` 偏写实摄影感，建筑棱角与街面秩序更清晰；`1.0` 则更偏奇幻插画氛围，远景层次、灯火密度与“热闹感”更强。`0.6` 与 `0.8` 处于两端之间，形成渐进过渡。

### 4.2 审美与真实感的取舍

从审美角度看，较高 LoRA 强度往往带来更统一的风格语言；但从结构真实性看，局部解剖稳定性并不一定随强度单调提升。A3/A4 及 B 组中出现的手足问题说明：风格强化并不等同于结构可靠。
因此，若目标是“稳定出片”，建议在 `0.6-0.8` 区间做精细调参；若目标是“风格强烈”，可使用 `1.0`，但应增加人工筛图与局部修复步骤。

### 4.3 小结

A 组结果表明，LoRA 在风格塑形方面具有明确效果，同时也暴露出“风格强度提升”与“结构稳定性”之间的平衡问题。这为后续 B/C 组分析提供了统一解释框架：生成结果由 `prompt` 信息量、LoRA 强度与场景难度共同决定，而非单一参数主导。

## 5. 结果二：Prompt 从简单到长文本（对应要求 2）

### 5.1 B 组设计回顾

B1/B2/B3 分别对应短、中、长 prompt（`B1_prompt_short_heroine`、`B2_prompt_medium_heroine`、`B3_prompt_long_heroine`），三者使用同一 seed（`971947498959982`），每组均测试 `0.0 / 0.6 / 0.8 / 1.0` 四档强度。该设计使“prompt 长度”成为主要变量，便于观察模型在信息约束变化下的响应。

### 5.2 主要观察

在本次对比中，随着 prompt 由短到长，同一子组内四档强度结果的构图与角色气质呈现出“更互像”的趋势；相反，短 prompt 下各强度图之间差异更大。可理解为：当语言约束较少时，LoRA 强度更容易主导视觉走向；当语言约束充分时，强度变化的自由度被部分压缩。
另外，B1/B2 的若干高强度档出现了手指问题，而 B3 未出现同等级明显问题。该现象可能与长 prompt 对局部结构约束更细有关，但当前证据仅来自单轮样本，不能直接推广为通则。

### 5.3 小结

B 组已完成“简单到长文本 prompt”的作业要求，并给出明确工程经验：长 prompt 并非必然更优，但在本次设置下可提升结果一致性与可控性。实际应用中需要在“描述成本”与“约束收益”之间进行平衡。

## 6. 不足与展望（对应要求 3）

### 6.1 失败样例与局限类型

根据问题样张索引，本次主要问题集中在三类：
1) **手部/足部结构问题**（A3、A4、B1、B2、C1）；
2) **多人动态交互不稳定**（C1 双人打斗场景）；
3) **文字可读性与风格化冲突**（C2 夜市多招牌）。

为保证正文与附录可逐图对应，明显问题样张按原始文件名列示如下（均为单轮、未复现）：

| C2 (`1.0`) | 字形变形 | `C2, 1082998128340460, 1.0, 1.0.png` |

其中 C1 的困难可由数据分布解释：训练集以单主体风格图为主，多人强动态属于 OOD（分布外）任务。C2 则体现出另一种张力：高强度风格下，字形更接近“奇幻装饰字”，视觉氛围提升，但与严格拼写可读性存在冲突。

### 6.2 工具链层面不足

- ComfyUI 节点自由度高，但学习与排错成本也较高；
- 训练环境对依赖版本较敏感，配置迁移成本不低；
- 个人作业条件下难以建立系统化评价（如多 seed、客观指标），结果更依赖人工审美判断。

这些问题并不否定工具价值，而是说明“开源高可控”与“工程复杂度”在同一流程中并存，需要通过流程设计加以平衡。

### 6.3 未来改进方向

后续若继续推进，优先级如下：
- **复现增强**：为 A/B/C 每组引入多 seed、多轮采样，降低偶然性；
- **数据扩展**：补充多人交互、复杂手势、文字招牌场景；
- **流程优化**：建立“自动筛选 + 人工复核”机制，减少手工挑图成本。

## 7. 结论（Conclusion）

本项目已完成既定核心任务：在 ComfyUI 工作流中完成自训练 LoRA，并围绕风格对比、prompt 复杂度和不足展望给出可复现实验记录。
在本次实验条件下，我确认：LoRA 强度能够显著塑造日系魔幻风格；长 prompt 在一定程度上提高了结果一致性；手部结构、多人动态与文字可读性仍是当前短板。
综合控制力、可复现性与扩展潜力，我将继续采用 `ComfyUI + Ostris + FLUX.1-dev` 的技术路线，并以多 seed 复现与数据覆盖扩展作为后续改进重点，以提升结论可靠性与模型泛化表现。
